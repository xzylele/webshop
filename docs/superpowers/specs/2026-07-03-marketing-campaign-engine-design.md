# Marketing Campaign Engine Design

วันที่: 2026-07-03

## เป้าหมาย

สร้างระบบแนะนำและจัดการแคมเปญการตลาดที่ช่วยดึงลูกค้าเดิมกลับมาซื้อซ้ำ พร้อมลดงานวิเคราะห์และติดตามผลของแอดมิน แอดมินยังเป็นผู้ตรวจและอนุมัติทุกแคมเปญก่อนส่ง รุ่นแรกส่งผ่านกล่องแจ้งเตือนภายในเว็บ และวางขอบเขตช่องทางให้รองรับ LINE หรืออีเมลในอนาคตโดยไม่ต้องเปลี่ยนแกนหลักของระบบ

## ขอบเขตรุ่นแรก

- เทมเพลตกลุ่มเป้าหมาย 3 แบบ: ลูกค้าที่ไม่ได้ซื้อเกิน 30 วัน, ลูกค้าที่ซื้อหมวดเดิมบ่อย และลูกค้า VIP
- ระบบคำนวณจำนวนผู้รับและต้นทุนส่วนลดสูงสุดโดยประมาณก่อนอนุมัติ
- แอดมินเลือกคูปอง แก้ข้อความ ดูตัวอย่าง อนุมัติ ส่งทันที หรือตั้งเวลาส่ง
- แจ้งเตือนลูกค้าภายในเว็บ พร้อมสถานะอ่านแล้ว/ยังไม่อ่านและลิงก์ปลายทาง
- รายงานการส่ง การเปิด การคลิก การใช้คูปอง ยอดขาย และ Conversion
- ลูกค้าปิดรับข้อความส่งเสริมการขายได้ โดยไม่กระทบข้อความธุรกรรมหรือความปลอดภัย

สิ่งที่ไม่อยู่ในรุ่นแรก ได้แก่ LINE, อีเมล, workflow หลายขั้น, A/B testing, การสร้างข้อความด้วย AI และการส่งแคมเปญโดยไม่ผ่านการอนุมัติของแอดมิน

## สถาปัตยกรรม

### Audience Analyzer

วิเคราะห์ผู้ใช้และรายการซื้อที่สำเร็จแล้วจากข้อมูลเดิม และคืนผลเป็นรายการผู้ใช้พร้อมเหตุผลที่เข้าเกณฑ์ หน่วยนี้ไม่ส่งข้อความหรือแก้ข้อมูลแคมเปญ จึงทดสอบกฎแต่ละแบบได้โดยอิสระ

กฎเริ่มต้น:

- `inactive_30_days`: เคยซื้อสำเร็จอย่างน้อยหนึ่งครั้ง แต่ไม่มีการซื้อสำเร็จใน 30 วันล่าสุด
- `frequent_category`: ซื้อสินค้าในหมวดเดียวกันสำเร็จอย่างน้อย 3 ครั้งภายใน 90 วัน
- `vip`: ยอดซื้อสำเร็จรวมตลอดอายุบัญชีอยู่ในกลุ่ม 10% แรกของลูกค้าที่เคยซื้อ โดยต้องซื้อสำเร็จอย่างน้อย 3 ครั้ง

เกณฑ์ตัวเลขถูกบันทึกใน `audience_rules` ของแคมเปญเพื่อให้ตรวจสอบย้อนหลังได้ แม้ค่าเริ่มต้นของเทมเพลตเปลี่ยนภายหลัง

### Campaign Manager

รับผิดชอบวงจรชีวิตแคมเปญ การตรวจข้อมูล การสร้าง snapshot ผู้รับ การอนุมัติ การตั้งเวลา และการยกเลิก สถานะที่อนุญาตคือ:

`draft -> approved -> scheduled -> sending -> completed`

แคมเปญที่ส่งทันทีเปลี่ยนจาก `approved` เป็น `sending` ได้โดยไม่ผ่าน `scheduled` สถานะ `cancelled` ใช้ได้ก่อนเริ่มส่ง และ `paused` ใช้เมื่อพบข้อผิดพลาดระหว่างการส่งหรือแอดมินหยุดรายการที่ยังไม่ส่ง

### Delivery Adapter

กำหนดสัญญากลางสำหรับการส่ง ได้แก่ channel, recipient, title, body, action URL และ metadata รุ่นแรกมี `in_app` adapter ซึ่งเขียนข้อมูลลงกล่องแจ้งเตือนลูกค้า การเพิ่ม LINE หรืออีเมลภายหลังต้องเพิ่ม adapter ใหม่โดยไม่แก้กฎกลุ่มเป้าหมายหรือวงจรชีวิตแคมเปญ

### Campaign Analytics

รวบรวม event และคำนวณผลลัพธ์ต่อแคมเปญ ตัวเลขสรุปต้องสร้างใหม่จากข้อมูลต้นทางได้ ไม่พึ่งค่ารวมสะสมเพียงอย่างเดียว

## แบบจำลองข้อมูล

### `campaigns`

เก็บชื่อ, template type, audience rules แบบ JSON, หัวข้อ, เนื้อหา, action URL, coupon ID, channel, เวลาเริ่มส่ง, สถานะ, ผู้สร้าง, ผู้อนุมัติ, เวลาอนุมัติ, เวลาเริ่ม/เสร็จ และ timestamps

### `campaign_recipients`

เก็บ campaign ID, user ID, เหตุผลที่เข้าเกณฑ์, สถานะการส่ง, delivery attempts, เวลาส่ง และ error ล่าสุด ใช้ unique constraint ที่ `(campaign_id, user_id, channel)` เพื่อป้องกันการส่งซ้ำ

รายชื่อผู้รับถูกสร้างเป็น snapshot เมื่อแอดมินอนุมัติ ก่อนสร้าง snapshot ระบบตัดผู้ที่ opt out และผู้ที่ได้รับแคมเปญการตลาดใน 7 วันล่าสุดออก

### `customer_notifications`

เก็บ user ID, campaign ID (nullable สำหรับข้อความที่ไม่ใช่การตลาด), ประเภทข้อความ, title, body, action URL, เวลาอ่าน และ timestamps ข้อความธุรกรรมกับข้อความการตลาดต้องมีประเภทแยกกัน

### `campaign_events`

เก็บ campaign ID, recipient ID, event type, transaction ID (ถ้ามี), metadata และเวลาที่เกิด event ประเภทเริ่มต้นคือ `sent`, `opened`, `clicked`, `coupon_redeemed` และ `purchased`

### `user_marketing_preferences`

เก็บ user ID, การยินยอมรับข้อความในเว็บ และเวลาแก้ไข ค่าเริ่มต้นอนุญาตข้อความในเว็บสำหรับสมาชิกเดิม แต่หน้าแจ้งเตือนต้องมีทางปิดรับที่มองเห็นได้ชัดเจน

## ลำดับการทำงาน

1. หน้าแนะนำเรียก Audience Analyzer เพื่อแสดงจำนวนลูกค้าในแต่ละเทมเพลต
2. แอดมินเลือกเทมเพลต คูปอง ข้อความ action URL และเวลาส่ง
3. ระบบแสดงตัวอย่างจำนวนผู้รับ สาเหตุที่เข้าเกณฑ์ และต้นทุนส่วนลดสูงสุดจากจำนวนผู้รับคูณส่วนลดสูงสุดต่อคน
4. เมื่อแอดมินอนุมัติ ระบบตรวจสิทธิ์ คูปอง วันหมดอายุ จำนวนสิทธิ์ งบประมาณ และสร้าง recipient snapshot ภายในธุรกรรมเดียวกัน
5. ตัวประมวลผลเลือกแคมเปญที่ถึงเวลาส่ง เปลี่ยนเป็น `sending` และส่งเป็นชุดผ่าน Delivery Adapter
6. การส่งแต่ละรายการใช้ unique key ป้องกันการสร้าง notification ซ้ำเมื่อ retry
7. เมื่อส่งครบ แคมเปญเปลี่ยนเป็น `completed`; หากเกิดข้อผิดพลาดที่กระทบทั้งแคมเปญให้เปลี่ยนเป็น `paused` และแจ้งแอดมิน
8. การเปิดหรือคลิกแจ้งเตือนสร้าง event โดย endpoint ที่ตรวจว่าผู้ใช้เป็นเจ้าของ notification
9. เมื่อใช้คูปองหรือซื้อสำเร็จ purchase flow บันทึก campaign event ภายในธุรกรรมเดียวกับคำสั่งซื้อ

## การวัดผล

- Delivery rate = ผู้รับที่ส่งสำเร็จ / ผู้รับทั้งหมด
- Open rate = ผู้รับที่เปิดอย่างน้อยหนึ่งครั้ง / ผู้รับที่ส่งสำเร็จ
- Click rate = ผู้รับที่คลิกอย่างน้อยหนึ่งครั้ง / ผู้รับที่ส่งสำเร็จ
- Coupon redemption rate = ผู้รับที่ใช้คูปอง / ผู้รับที่ส่งสำเร็จ
- Conversion rate = ผู้รับที่ซื้อสำเร็จโดยใช้คูปองของแคมเปญ / ผู้รับที่ส่งสำเร็จ
- Campaign revenue = ยอดสุทธิของธุรกรรมสำเร็จที่ใช้คูปองของแคมเปญ
- Returning customers = ผู้รับกลุ่ม inactive ที่ซื้อสำเร็จโดยใช้คูปองก่อนวันหมดอายุ

รุ่นแรกใช้การใช้คูปองเฉพาะแคมเปญเป็นเกณฑ์ attribution เพื่อให้ตัวเลขชัดเจน ไม่อ้างรายได้จากการคลิกเพียงอย่างเดียว

## หน้าจอ

### หลังบ้าน

- ภาพรวม: รายได้จากแคมเปญ, Conversion, จำนวนคูปองที่ใช้ และลูกค้าที่กลับมาซื้อ
- คำแนะนำ: การ์ด 3 เทมเพลต พร้อมจำนวนผู้รับ เหตุผล และปุ่มสร้างแคมเปญ
- ตัวสร้าง: ฟอร์มข้อความ คูปอง action URL เวลาส่ง ตัวอย่าง notification และประมาณการต้นทุน
- ประวัติ: ค้นหา/กรองตามสถานะ พร้อมผู้สร้าง ผู้อนุมัติ ผลลัพธ์ และข้อผิดพลาด
- รายละเอียด: funnel ตั้งแต่ผู้รับจนถึงการซื้อ พร้อม audit trail

### ลูกค้า

- กระดิ่งแสดงจำนวนที่ยังไม่อ่าน
- รายการแจ้งเตือนรองรับอ่านทีละรายการและอ่านทั้งหมด
- การคลิกพาไปยัง action URL ที่ระบบยอมรับเท่านั้น
- หน้าตั้งค่ามีตัวเลือกเปิด/ปิดข้อความส่งเสริมการขาย

## สิทธิ์ ความปลอดภัย และความทนทาน

- API จัดการแคมเปญทุกเส้นทางตรวจบทบาทแอดมินบนเซิร์ฟเวอร์
- URL ปลายทางต้องเป็น path ภายในร้านหรือโดเมนที่อยู่ใน allowlist เพื่อป้องกัน open redirect
- หลังอนุมัติห้ามแก้ audience rules, coupon, channel และข้อความ หากต้องเปลี่ยนให้ยกเลิกและสร้างฉบับใหม่
- ก่อนเริ่มส่งตรวจว่าคูปองยังใช้งานได้ วันหมดอายุอยู่หลังเวลาส่ง และ quota เพียงพอตามนโยบายที่แอดมินกำหนด
- การส่งเป็นชุด รองรับ retry แบบมีเพดาน และไม่ส่งรายการที่สำเร็จแล้วซ้ำ
- เก็บ audit trail สำหรับการสร้าง แก้ไข อนุมัติ ยกเลิก หยุด และเริ่มต่อ
- จำกัดลูกค้าหนึ่งคนให้รับแคมเปญการตลาดไม่เกินหนึ่งแคมเปญในช่วง 7 วัน โดยตรวจตอนสร้าง snapshot
- การ opt out มีผลกับรายการที่ยังไม่ส่ง แม้อยู่ใน snapshot แล้ว ตัวส่งต้องตรวจ preference ซ้ำก่อนส่ง

## การตั้งเวลา

ระบบใช้ endpoint สำหรับประมวลผลงานตามเวลา ซึ่งเรียกได้จาก scheduler ภายนอกที่มี shared secret และเรียกซ้ำได้อย่างปลอดภัย การเรียกหนึ่งครั้ง claim งานจำนวนจำกัดเพื่อหลีกเลี่ยง timeout และการแข่งขันระหว่าง worker รายละเอียด API จะยึดตามคู่มือ Next.js 16.2.9 ที่อยู่ใน `node_modules/next/dist/docs/` ก่อนเริ่ม implementation

## การทดสอบและเกณฑ์ยอมรับ

- Unit tests ครอบคลุมกฎผู้ใช้ทั้ง 3 กลุ่ม รวม boundary ของ 30/90 วันและ percentile VIP
- Integration tests ครอบคลุมสิทธิ์แอดมิน การสร้าง snapshot การ opt out frequency cap และ transition ทุกสถานะ
- ทดสอบ retry แล้วไม่เกิด notification หรือ event `sent` ซ้ำ
- ทดสอบคูปองหมดอายุ ถูกปิด หรือ quota ไม่พอแล้วแคมเปญไม่เริ่มส่ง
- ทดสอบว่าผู้ใช้เปิด/คลิก notification ของผู้อื่นไม่ได้
- ทดสอบ purchase flow แล้ว campaign attribution ถูกบันทึกครั้งเดียวในธุรกรรมเดียวกัน
- ทดสอบตัวเลข Dashboard โดยคำนวณจาก fixture ที่ทราบผลล่วงหน้า
- End-to-end flow: แอดมินสร้างและอนุมัติแคมเปญ, scheduler ส่ง, ลูกค้าเปิดและใช้คูปองซื้อ, Dashboard แสดง conversion และ revenue ถูกต้อง

ระบบถือว่าพร้อมใช้งานเมื่อทุกเกณฑ์ข้างต้นผ่าน, แอดมินสามารถทำ flow หลักโดยไม่แก้ฐานข้อมูลด้วยตนเอง และการเรียก scheduler ซ้ำไม่สร้างข้อความหรือยอดรายงานซ้ำ
